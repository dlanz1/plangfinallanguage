Golf-themed static language

3DTee

A programming language where source code plays like a round of golf: start at teeOff, finish at clubHouse, keep values in bags, pin constants in place, and let the compiler keep score.

View Compiler Read Examples Open Grammar

The Story

3DTee turns a golf scorecard into a programming model. A program opens with teeOff, closes with clubHouse, and uses domain language like bag, pin, swing, readLie, and play to make the code feel like moving through a course.

The language is designed to be playful without giving up rigor. It has static types, arrays, optionals, structured records, functions, loops, scoped blocks, returns, breaks, and expression forms rich enough to model scorecards, leaderboards, tee times, and course statistics.

The compiler follows the classic course pipeline: Ohm grammar, parser, analyzer, optimizer, and JavaScript generator. All four stages are implemented and the test suite reports 100% line, statement, function, and branch coverage on every source file.

Language Features

Program Shape

Every complete program starts with teeOff and ends with clubHouse.

Typed Declarations

Use bag for variables, pin for constants, and course for record-like types.

Functions

Declare typed functions with swing, parameters, optional return annotations, and sink returns.

Control Flow

Branch with readLie, loop with whileBall, practice, and play, and exit loops with shank.

Optionals

Model missing values with T?, hazard T, null-coalescing ??, loft, and bounce.

Expressions

Supports arithmetic, comparison, logic, bitwise operations, shifts, exponentiation, ternaries, calls, indexing, and course member access.

Example Programs

Scorecard

records, arrays, collection loop 
teeOff
  course HoleScore {
    hole: int;
    strokes: int;
  }

  bag totals: [int] = [3, 4, 5, 4];
  bag backups: [int] = [int]();
  bag holes = #totals;
  bag first = totals[0];

  play s through totals {
    print(s);
  }
clubHouse

Handicap

functions, constants, assignment, if 
teeOff
  pin par: int = 72;
  bag score: int = 74;

  swing relativeToPar(s: int): int {
    sink s - par;
  }

  bag delta = relativeToPar(score);
  readLie delta <= 0 {
    print("under or even");
  } otherwise {
    score = score - 1;
  }
clubHouse

Leaderboard

while, practice, break 
teeOff
  bag strokes = 0;

  whileBall strokes < 5 {
    strokes = strokes + 1;
    readLie strokes == 3 {
      shank;
    }
  }

  practice {
    strokes = strokes - 1;
  } whileBall strokes > 0;
clubHouse

Caddie

optionals and function types 
teeOff
  swing applyFn(f: (int) -> int, x: int): int {
    sink f(x);
  }

  bag maybeScore: int? = hazard int;
  bag hasScore = loft maybeScore;
  bag score = maybeScore ?? 72;
  bag realScore = bounce maybeScore;
clubHouse

operators and numeric literals

Lie Check

conditionals, booleans, blocks 
teeOff
  pin par = 72;
  bag score = 70;
  bag underPar = score < par;

  readLie underPar {
    bag message = "attack the pin";
  } otherwise {
    bag message = "play safe";
  }
clubHouse

Try It

Clone the repo, install dependencies, and pick a 3DTee program and an output type to see each stage of the pipeline in action: 

git clone https://github.com/dlanz1/3DTee.git
cd 3DTee
npm install

# Confirm grammar match only
node src/3DTee.js examples/handicap.3dt parsed

# Inspect the analyzed AST
node src/3DTee.js examples/handicap.3dt analyzed

# Inspect the optimized AST
node src/3DTee.js examples/handicap.3dt optimized

# Emit JavaScript
node src/3DTee.js examples/handicap.3dt js

The last command, run on examples/handicap.3dt, produces: 

function relativeToPar(score, par) {
  return (score - par);
}
let today = relativeToPar(74, 72);

Static, Safety, and Security Checks

The analyzer enforces every rule below. Violations are surfaced as CompileUserError messages that include the offending line and column.

  • An identifier cannot be redeclared in the same scope.
  • An identifier must be declared before it is used.
  • Variable, parameter, and field types must resolve to a known type.
  • An initializer must be assignable to its declared (or inferred) type.
  • int widens to float on assignment; otherwise types must match exactly. Function types match by arity and component-wise assignability.
  • The target of an assignment must be mutable: a bag, a parameter, an element of either, or a field of either — never a pin.
  • Arithmetic operators require numeric operands; + additionally accepts two strings.
  • Bitwise (|, ^, &) and shift (<<, >>) operators require integer operands.
  • Logical operators (&&, ||, !) and the tests in readLie, whileBall, practice, and ternaries must be boolean.
  • Relational ordering (<, <=, >, >=) requires numeric or string operands; both sides of any comparison must have the same type.
  • Array indexing requires an array on the left and an integer on the right; # requires an array.
  • Member access requires a course value and a field name declared on that course; field names must be distinct.
  • A function call's argument count must match the declared arity, and each argument must be assignable to its parameter type.
  • loft and bounce require an optional; ?? requires an optional on the left and a base-type-compatible value on the right.
  • shank may appear only inside a loop; sink may appear only inside a function, and a value-sink requires a non-void return type.
  • Array literals must be homogeneous; the empty bracketed literal [] infers as [any], and a typed empty array uses [T]().
  • Unicode escapes inside strings must be valid code points (≤ 0x10FFFF).
